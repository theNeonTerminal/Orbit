Brief Technical Working: ESP32 streams Microphone audio to server and server processes the audio input and sends back an audio output along with

Client:

Orbit Hardware:
- ESP32 at its core
- INMP441 I2S Mic
- MAX98357A Amplifier for speaker
- touch sensor x1
- IR proximity sensor x2
- Servo Motor x2
- L293D Motor Driver x1
- 0.96" SSD1306 OLED display

Orbit is a physical presence of a remote server, which receives audio input from orbit, processes it and sends back audio data along with a JSON object which will be interpreted by orbit and call functions accordingly.

ESP32 and Websockets part:
Whenever Orbit is booted up, it opens up a new connection with the server. Now Orbit/Client will stream audio data to server for the time it is turned on, NO PROCESSING HAPPENS ON BOARD WITH ESP32, ALL THE AUDIO PROCESSING PART TAKES PLACE IN THE SERVER. 
The ESP32 always streams the audio data form the I2S Microphone to the server as long as it is powered on. The server runs on python an uses a porcupine wake word detection to listen for the wake word "Orbit" or any other word which it is configured to listen for. The server once detects the wake word, will send a JSON object to the ESP32 to display the text "Listening..." or any other configured word on the OLED display. For more details, refer to server working for this part. Once the server receives "stop" signal, it will then send another JSON object to display the transcribed text on the OLED display. Now the server processes the transcribed text and returns a JSON object along with a .wav audio file streamed which the ESP32 will catch like an internet radio and stream it via the speaker connected to the MAX98357A amplifier. The OLED also shows the output text or call a function if input from server via JSON Object is received.

ESP32 and Sensors/display/motors:
ESP32 runs all the displays (OLED SSD1306), motors (servos and motors connected to L293D) and sensor data (IR proximity, Touch sensor) are processed by ESP32 on board.

1. IR proximity sensors for drop detection, when high (ie: floor/desk exactly below robot) move normal, when LOW (ie: nothing below robot) Stop the robot from moving only in that detection
There  will be 2 IR proximity sensors, 1 connected in front and one in back, both will be HIGH, when floor/desk is detected then stop the direction motor from moving
2. OLED display: To display text, emotions from roboEyes library and other custom made animations
3. Servo: 2 servos one connected to its arm, to be able to express its emotions and one connected to its head to able to rotate his head on the y axis, not x or z
4. Motor Driver to Control motors
5. Mic and Speaker already explained

ESP32 functions:
1. display_text(str: text): it receives one argument and simply prints out the text on a new line on the OLED display.
2. display_time(int: time_offset): this function simply fetches the time from the internet for a given place, time_offset in seconds is given by gemini via JSON for the appropriate place.
3. set_timer(int: mins, int: secs): User can set a max time of 1hour, this function is special because it is asynchronous and the user can again ask the assistant while the timer runs, the following 2 functions help it to properly function. After the timer ends, the ESP32 will send signal to serve and server will stream beep.wav to the ESP32 so indicate the timer has ended and then again return to the robo eyes. This function must be optimized in code to take in account for the other 2 supplement functions given.
4. pause_timer(bool: paused): User can ask ai assistant to Pause the timer OR use the built in touch sensor to pause the timer, simply invert the state of the timer, 1 means paused and 0 means running.
5. runTimer_bg(): This function takes no argument, and simply runs the timer in the background to free the OLED display to display something else (when the user asks something while the timer is running then this function will be executed to follow the steps: display Listening and then the users prompt and the output given by the model).
6. show_weather(int: temp, bool: hot/cold, bool: unit): This is a custom function created by the user which displays the weather with cool graphics. bool hot or cold: 0 is hot, 1 is cold (threshold: 25 deg C). Unit is 0: Celsius (default) 1: Fahrenheight, bool because its easier to process than strings and we have only 2 values here

Emotions shown by Orbit based on situation:
1. Normal: Normal rectangular/square eyes with plain face.
2. Thinking: A thinking face or some graphics which convey the message "processing"/"thinking"
3. Bored: Waiting to get pat on the touch sensor.
4. Happy: Shown once pat on the touch sensor.

Sensor Integration:
1. IR sensors: Anti Drop sensors, when high, desk in underneath, keep moving, once LOW, immediately stop motors to prevent fall.
2. Touch Sensor: Deactivated during a processing, like thinking or speaking output. Once hihg, Orbit shows Happy emotion and thats it
3. Bored: Waiting to get pat on the touch sensor.
4. Happy: Shown once pat on the touch sensor.

NOTE: For these, the OLED display must be updated first before the audio is spoken out, so that the user can follow the OLED and audio simultaneously
Audio FORMAT: PCM, 16-bit, mono, 16 kHz, little endian, 1024-sample chunks

State Of the Machine:
IDLE (wakeword detection) -(Wake word detected)-> Server: sends JSON to client to stop moving and change OLED text to listening -(max 10s or 5s silence)-> STT is performed by server & Gemini API, JSON sent to Client to change text to input given -> Server & Gemini APi Processes transcribed text and send back a JSON object to client along with .wav audio streaming captured by client -> IDLE (wakeword detection)

FOR ESP32:
Normally it would simply roam around the place, while following IR sensor readings and sending mic data to server, then the server does its job of wakeword detection.

Server:
Python will handle server side stuff. Servers only task is to handle websockets connections, send data to Gemini API, audio and AI stuff. 

JSON Object to be sent immediately after wake word:
{
  "type": "wakeword_detected",
  "data": {
    "display_text": "Listening..."
  }
}

JSON to be called after transcription:
{
  "type": "transcription",
  "data": {
    "prompt": "Who is the Prime Minister of India?"
  }
}

JSON to be called after the server processes the transcribed text and return resut:
{
  "type": "response",
  "data": {

  }
}

Note: There WILL BE A SPECIFIED number of hardcoded additional functions to sometimes call, say for instance if the user asks for weather, then the client/Orbit will execute weather display function which will contain some graphics, similarly for a timer or other things. The audio .wav file WILL NOT BE TRANSMITTED via JSON, it will streamed on the websockets connection which will be played by the ESP32 in real time.

The server will simply handle all the websockets communication and the hard part.
Server working in brief: Server opens a connection whenever the ESP32 boots up, starts a full duplex communication channel and always listens to the I2S audio given by the ESP32. The server will be in low power mode most of the time to save resources running Porcupine, once the wake word is detected, the server runs the function that 1st sends JSON data to change the eyes on the ESP32 display to "Listening" text, start transcribing the user given audio and then sends JSON to the client/ESP32 which displays the transcribed text, which is then sent to the Gemini API which then produces a result along with the audio TTS generated. The server will then again go to low power mode which will start Porcupine and then again listen for wake word. 

NOTE: All audio processing will be handled by GEMINI audio processing as given in the DOCS to refer.

Error Handling:
The OLED display on the client clearly shows Connecting to Wifi, and then Connection to server, if both are successful, then it runs, if any fails, it should show in the display that which one failed, and instruct the user to restart orbit (given code is correct, there still could be issues)

If the STT or TTS returns gibberish or output at all, then the user will be shown it, for all cases, before asking the gemini API for the transcribed text, users will have a 3s time to view the transcribed data and touch the touch sensor to discard the reading, here the server send the JSON to take orbit back to IDLE (usually sent after successful voice output ends) and orbit returns to normal mode giving the user a seconds chance to dictate.
In case if the audio is not transmitted effectively, then orbit simply shows the text and not speaks out loud for the output

If connection is lost Mid way, do the same, show the User that connection is lost and reboot the system.
